The normal Wix CLI workflow is easier to remember when you stop treating commands as isolated facts.

## The sequence

Study this until it feels automatic:

```text
create -> login -> dev -> generate -> code -> build -> preview -> release
```

That is the full loop.

Not every session needs every step, but almost every meaningful delivery path fits inside this sequence.

## What most daily work looks like

Most real sessions are shorter than the full loop:

```text
dev -> code -> test -> repeat
```

The later steps matter when you want a stable preview, a handoff build, or actual extension registration.

## Why preview is not release

This is one of the biggest practical distinctions in the handbook.

`preview` uploads and hosts the build so you can test it.

`release` also handles platform registration behavior that some extensions depend on.

This matters most when:

- a site widget feels half-wired
- an embedded script behaves differently than expected
- a generated extension exists locally but is not showing up the way you expect inside Wix

When something looks strange after a successful preview, release is often the next thing to check.

## Minimal healthy loop

```bash
wix login
wix dev
wix generate
wix build
wix preview
wix release
```

Again, this is not the sequence for every single coding session. It is the sequence you should understand
well enough to diagnose where you are in the lifecycle of the feature.

## Operational mindset

Treat these stages as different questions:

- `dev`: Can I build and iterate locally?
- `preview`: Can other people test this build?
- `release`: Has Wix fully recognized and registered the pieces that need platform registration?

That mental separation prevents a lot of confusion.
