The Wix Design System is a React component library for building dashboard extensions that feel native inside the Wix admin. If you're building dashboard pages, you should use it. If you're building site extensions, you have more freedom.

## The rules, depending on where you're building

<Tabs>
	<TabItem label="Dashboard">
		Dashboard extensions live inside the Wix admin shell. They should feel **operational** -- calm, predictable, clear.

		Good defaults:
		- Predictable page structure with `<Page>`, `<Page.Header>`, `<Page.Content>`
		- Legible labels and clear grouping
		- Steady spacing (the design system handles this)
		- Strong distinction between primary and secondary actions

		Use the Wix Design System components. Don't fight the built-in typography. Don't apply global CSS resets.

		```jsx
		import { Page, Card, Button } from 'wix-design-system';

		export default function SettingsPage() {
		  return (
		    <Page>
		      <Page.Header title="Settings" />
		      <Page.Content>
		        <Card>
		          <Card.Header title="General" />
		          <Card.Content>
		            <Button>Save Changes</Button>
		          </Card.Content>
		        </Card>
		      </Page.Content>
		    </Page>
		  );
		}
		```
	</TabItem>
	<TabItem label="Site">
		Site extensions have more visual freedom because they belong to the live site, not the admin shell.

		That doesn't mean you should ignore structure. It means your visual system should fit the **site experience** rather than mimic admin tooling.

		You can use your own component library, your own CSS, your own fonts. Just make sure the result feels like it belongs on the site.

		```jsx
		export default function PricingWidget() {
		  return (
		    <div className="my-pricing-widget">
		      <h2>Choose a Plan</h2>
		      {/* Your own components, your own styles */}
		    </div>
		  );
		}
		```
	</TabItem>
</Tabs>

## Why fonts break

Font problems almost always come from one of two causes:

<Steps>

1. **Loading a custom font incorrectly** -- Use `@font-face` or a hosted stylesheet, not a font that may not exist in the visitor's browser.

2. **Fighting the design system's typography** -- Dashboard pages inherit fonts from the Wix admin shell. A global `font-family` override on `body` or `*` will break every design system component on the page.

</Steps>

:::caution
Avoid blanket `font-family` overrides in dashboard extensions. If you need a custom font for a specific element, scope it tightly.
:::

```css
/* Safe -- scoped to your element */
.my-branded-header {
  font-family: 'Your Custom Font', sans-serif;
}

/* Dangerous -- breaks design system components */
body {
  font-family: 'Your Custom Font', sans-serif;
}
```

## Why layouts break

Layout problems usually trace back to one of these:

1. **Global CSS resets** that collide with the design system's spacing and box model.
2. **Assuming full viewport control** when your dashboard page runs inside an iframe or constrained panel.
3. **Mixing layout systems** -- applying CSS Grid to a container whose children expect flex-based flow.

:::note
The Wix admin shell controls the outer page chrome. Your extension controls only the content area inside it. Fixed positioning, `100vh` heights, and global margin resets will fight the shell.
:::

```css
/* Works -- uses the space you're given */
.my-dashboard-content {
  display: flex;
  flex-direction: column;
  gap: 16px;
  padding: 24px;
}

/* Fights the shell -- don't do this */
.my-dashboard-content {
  position: fixed;
  top: 0;
  left: 0;
  width: 100vw;
  height: 100vh;
}
```
